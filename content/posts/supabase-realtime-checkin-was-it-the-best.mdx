# Supabase Realtime으로 체크인 해결, 과연 최선이었을까?

---

## 도전 과제

300명 규모의 컨퍼런스 체크인 시스템을 1개월 안에 만들어야 했다. 핵심 제약은 두 가지였다.

**첫째, PC와 모바일이라는 두 디바이스를 실시간으로 연결해야 했다.**

모바일에서 QR을 스캔하면 PC 대시보드에 즉시 반영되어야 하고, PC에서 체크인을 처리한 결과가 모바일에 즉시 돌아와야 한다. 단순히 "서버가 클라이언트에 알려주는" 단방향이 아니라, 양쪽이 서로 메시지를 주고받는 **양방향 실시간 통신**이 필요했다.

**둘째, 행사 당일에는 실패가 곧 사고다.**

Wi-Fi가 끊기거나 세션이 만료되거나, 한쪽 디바이스만 연결된 상태에서 스캔하는 경우—이런 엣지 케이스를 모두 대비해야 했다. 300명이 줄 서 있는 상황에서 시스템이 멈추면 복구할 시간이 없다.

이 두 가지를 충족하는 실시간 통신 기술을 골라야 했고, 선택지는 생각보다 많았다.

---

## 선택지 비교 — 5가지 후보

### 1. HTTP Polling

가장 원시적인 방법. 클라이언트가 서버에 "변경됐어?" 하고 주기적으로 물어보는 것이다.

- **장점**: 구현이 가장 단순하다. 기존 REST API에 엔드포인트 하나 추가하면 끝
- **단점**: 주기(예: 1초)만큼 지연이 발생하고, 300명이 동시에 폴링하면 서버 부하가 급증한다
- **탈락 이유**: 체크인 현장에서 1초 지연은 "스캔했는데 반응이 없네?" → 다시 스캔 → 중복 처리로 이어진다. 양방향 통신도 자연스럽지 않다

### 2. SSE (Server-Sent Events)

HTTP 연결을 유지한 채 서버가 클라이언트에 일방적으로 데이터를 밀어주는 방식이다.

- **장점**: HTTP 기반이라 프록시/방화벽 호환성이 좋다. 브라우저 내장 `EventSource` API로 재연결이 자동 처리된다
- **단점**: **서버 → 클라이언트 단방향**이다. 모바일 → PC로 스캔 데이터를 보내려면 별도 REST API가 필요하고, 그 결과를 다시 모바일로 보내려면 또 SSE 채널이 필요하다
- **탈락 이유**: "모바일이 스캔 → PC가 처리 → 모바일에 결과 전송"이라는 왕복 흐름을 SSE + REST로 구현하면 두 가지 통신 계층을 관리해야 한다. 복잡도 대비 이점이 없었다

### 3. WebSocket 직접 구축

WebSocket 서버를 직접 띄워서 양방향 통신을 구현하는 방법. 가장 유연하고, 가장 많은 제어권을 가진다.

- **장점**: 프로토콜 레벨에서 완전한 제어가 가능하다. 메시지 포맷, 인증, 라우팅 등 모든 걸 원하는 대로 설계할 수 있다
- **단점**: 서버 인프라 셋업, 연결 관리(누가 접속 중인지, 끊겼는지), 스케일링(다중 서버 시 메시지 브로커 필요), 재연결 로직을 모두 직접 구현해야 한다
- **탈락 이유**: 1개월 안에 체크인 시스템 전체(신청, 폼, 체크인, 어드민)를 만들면서 WebSocket 서버의 안정성까지 검증하기에는 시간이 부족했다

### 4. Firebase Realtime Database

Google의 실시간 데이터베이스. 데이터가 변경되면 연결된 클라이언트에 자동으로 push한다.

- **장점**: SDK가 성숙하고 오프라인 동기화를 지원한다. 데이터가 자동으로 영속된다
- **단점**: 데이터 영속 위주로 설계되어 있어, 순수 메시징에는 오버스펙이다. 프로젝트가 이미 Supabase를 쓰고 있어서 Firebase를 추가하면 인프라 의존성이 이중으로 늘어난다. 가격도 연결 수 + 데이터 전송량 기반이라 예측이 어렵다
- **탈락 이유**: 체크인 데이터는 이미 Supabase DB에 저장하고 있었다. 실시간 통신만을 위해 Firebase를 별도로 붙이는 건 과잉이었다

### 5. Supabase Realtime Broadcast

Supabase에서 제공하는 Broadcast 모드. WebSocket 위에서 동작하지만, DB에 데이터를 저장하지 않고 **순수 메시징**만 수행한다.

- **장점**: 이미 프로젝트에서 Supabase를 사용 중이라 추가 인프라가 없다. 채널 기반으로 세션 격리가 자연스럽다. SDK가 연결 관리를 대신 해준다
- **단점**: at-most-once 전달이라 메시지 유실 가능성이 있다. Supabase 서비스 장애 시 전체 체크인이 멈춘다
- **선택 이유**: 기존 인프라 활용, 순수 메시징이라는 요구에 정확히 부합, 2-3일이면 핵심 구현이 가능

### 비교 요약

| | Polling | SSE | WebSocket 직접 | Firebase | Supabase Broadcast |
|---|---|---|---|---|---|
| 통신 방향 | 단방향 (요청-응답) | 단방향 (서버→클라) | 양방향 | 양방향 | 양방향 |
| 추가 인프라 | 없음 | 없음 | 서버 필요 | Firebase 프로젝트 | 없음 (기존 Supabase) |
| 실시간성 | 폴링 주기에 의존 | 실시간 | 실시간 | 실시간 | 실시간 |
| 구축 시간 | 1-2일 | 3-5일 | 2-3주 | 1주 | 2-3일 |
| 메시지 보장 | API 응답으로 확인 | 재전송 내장 | 직접 구현 | 자동 동기화 | 보장 없음 (at-most-once) |
| 데이터 저장 | 별도 구현 | 별도 구현 | 별도 구현 | 자동 저장 | 저장 안 함 |

## 수행 — Broadcast 기반 설계

### 채널과 세션의 분리

Supabase Broadcast의 채널 개념을 세션 단위로 활용했다. 채널명에 고유 세션 ID를 포함시켜, 여러 행사가 동시에 진행되어도 메시지가 섞이지 않도록 격리했다.

세션은 24시간 유효하게 설정하고, 로컬 스토리지에 저장하여 페이지 새로고침 시에도 기존 세션을 재사용할 수 있게 했다.

### 이벤트 프로토콜 설계

PC와 모바일 간에 주고받는 이벤트를 명확한 프로토콜로 정의했다.

| 이벤트 | 방향 | 용도 |
|--------|------|------|
| `ping` | 모바일 → PC | PC 연결 상태 확인 요청 |
| `pong` | PC → 모바일 | PC 연결 확인 응답 |
| `connected` | 모바일 → PC | 모바일 디바이스 접속 알림 |
| `scanned` | 모바일 → PC | QR 코드 데이터 전송 |
| `result` | PC → 모바일 | 체크인 처리 결과 |
| `expired` | PC → 모바일 | 세션 만료 알림 |
| `new_session` | PC → 모바일 | 새 세션 리다이렉트 안내 |

이 프로토콜이 시스템의 근간이다. 단순히 "데이터를 보낸다"가 아니라, **양쪽 디바이스의 상태를 동기화하는 상태 머신**을 설계한 것에 가깝다.

### Ping-Pong 헬스체크

Broadcast의 가장 큰 약점은 **수신자가 없어도 전송이 성공으로 처리된다**는 점이다. 모바일에서 QR을 스캔했는데 PC가 꺼져 있으면? 메시지는 허공으로 사라진다.

이를 해결하기 위해 모바일이 채널에 접속하면 가장 먼저 PC에 `ping`을 보내도록 했다. 5초 내에 `pong`이 돌아오지 않으면 "PC가 연결되어 있지 않습니다"라는 안내를 표시하고, **스캔 자체를 차단**한다. 메시지 유실이 발생할 수 있는 구간에서는 아예 전송이 일어나지 않도록 원천 방지한 셈이다.

### 모바일 상태 머신

모바일 스캐너는 단순한 카메라 앱이 아니라, 명확한 상태 전이를 따르는 상태 머신으로 설계했다.

```
loading → waiting ──(pong 수신)──→ ready
             │                       │
             │(5초 타임아웃)           │(QR 스캔)
             ↓                       ↓
           no_pc                  scanning
                                     │
                               ┌─────┴─────┐
                               ↓           ↓
                            success      error
                               │           │
                            (3초 후)     (3초 후)
                               └──→ ready ←┘

                     expired ──(자동 새로고침)──→ loading
```

각 상태에서 허용되는 액션이 정해져 있어, **비정상적인 순서의 조작이 원천 차단**된다. `ready` 상태가 아니면 카메라가 활성화되지 않는다.

### 세션 만료와 자동 복구

24시간 세션이 만료되면 PC가 자동으로 새 세션을 생성하고, 기존 채널을 통해 모바일에 새 세션 ID를 알려준다. 모바일은 이를 받아 자동으로 새 세션 URL로 리다이렉트한다.

스태프가 세션 만료를 인지하지 못해도 시스템이 스스로 복구하는 구조다. 행사 운영 중에 "세션 갱신해 주세요"라고 안내해야 하는 상황 자체를 없앤 것이다.

### 폴백 모드

Supabase 서비스 자체에 장애가 발생하면? 이 경우를 위해 PC에서 직접 바코드 스캐너로 체크인하는 대체 모드를 구현했다. 모바일 없이 PC 단독으로 체크인을 처리할 수 있어, 실시간 통신 계층이 완전히 죽어도 체크인 업무는 계속된다.

## 결과

- **300명 규모 행사에서 무장애 운영** 달성
- 스캔 → 체크인 완료까지 Express 모드 기준 **1초 이내** 처리
- ping-pong 헬스체크로 PC-모바일 연결을 사전 검증하여 **스캔 실패율 0%**
- 세션 만료 자동 갱신으로 스태프 개입 없이 **24시간 연속 운영** 가능
- Broadcast 기반 설계부터 구현까지 **2-3일**에 핵심 기능 완성, 전체 시스템은 1개월 내 배포

## 후기 — 과연 최선이었을까?

### 잘한 점

**기존 인프라를 활용한 판단**이 가장 컸다. 프로젝트가 이미 Supabase를 쓰고 있는 상황에서, 실시간 통신을 위해 새로운 서비스를 추가하는 건 리스크를 늘리는 행위였다. Supabase Broadcast는 SDK 하나로 바로 붙일 수 있었고, 이 덕분에 비즈니스 로직—체크인 처리, 세션 관리, 상태 전이 설계—에 시간을 집중할 수 있었다.

**Broadcast의 약점을 애플리케이션 레벨에서 보완한 것**도 좋은 판단이었다. at-most-once 전달이라는 근본적 한계를 ping-pong 헬스체크로 우회한 덕분에, 메시지 유실 가능성이 있는 구간에서는 전송 자체가 일어나지 않았다.

### 아쉬운 점

**디버깅이 어려웠다.** Broadcast 메시지는 DB에 남지 않기 때문에, 개발 중에 "이 메시지가 왜 안 도착했지?"를 추적하기가 까다로웠다. 콘솔 로그에 의존할 수밖에 없었고, 프로덕션에서도 중요 이벤트의 흔적이 남지 않아 사후 분석이 힘들었다. 돌이켜보면 주요 이벤트를 별도로 로깅하는 레이어를 초기에 구축했어야 했다.

**오프라인 시나리오에 대한 대비가 부족했다.** Wi-Fi가 완전히 끊기면 Broadcast 자체가 불가능한데, 이때 모바일에서 스캔한 데이터를 로컬에 큐잉했다가 연결 복구 시 일괄 전송하는 기능까지는 구현하지 못했다. 300명 규모에서는 문제가 없었지만, 야외 행사나 네트워크 인프라가 열악한 환경이라면 반드시 필요한 기능이다.

### 다시 한다면?

규모와 기한이 같다면 **같은 선택을 할 것이다.** 300명, 1개월이라는 조건에서 Supabase Broadcast는 최소 투자로 최대 효과를 낸 선택이었다.

하지만 조건이 바뀐다면 이야기가 달라진다.

- **1,000명 이상, 다중 현장 동시 운영**: 동시 접속 수가 Supabase 요금 플랜에 묶이고, 채널 수 폭증 시 관리가 어려워진다. 이 규모에서는 Socket.IO + Redis Adapter나 자체 WebSocket 서버를 고려할 것이다.
- **메시지 유실이 절대 불가한 시스템** (예: 결제 연동 체크인): Broadcast 대신 Supabase의 Postgres Changes(DB 변경 기반 구독)나 메시지 큐(RabbitMQ, Redis Streams)를 쓸 것이다. 메시지가 DB에 영속되므로 유실 가능성이 원천 차단된다.
- **기한이 3개월 이상**: WebSocket 서버를 직접 구축하는 것도 선택지에 들어온다. 프로토콜 레벨에서 완전한 제어권을 갖는 것의 가치가 구축 비용을 상쇄한다.

기술 선택에 절대적인 "최선"은 없다. **그 시점의 제약 조건 안에서 합리적인 판단**이 있을 뿐이다. Supabase Broadcast는 300명 규모, 1개월 기한, 기존 Supabase 인프라라는 세 가지 조건이 만들어낸 합리적인 선택이었고, 그 조건 위에서 충분히 잘 작동했다.

중요한 건 기술 자체가 아니라, **왜 이 기술을 골랐고, 무엇을 포기했으며, 그 포기를 어떻게 보완했는지**를 설명할 수 있느냐다.
